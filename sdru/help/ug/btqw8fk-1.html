Determining If You Need Burst Mode

comm.SDRuReceiver System Object

Set the System objectTM OverrunOutputPort property to true so that the step method outputs a value to indicate if one or more packets is dropped during host reception of USRP® hardware data.

  • Zero indicates no data loss

  • A positive number indicates that overruns occurred

The default value for OverrunOutputPort is false, which means that the port is not enabled and no information about dropped packets is displayed.

This port is a useful diagnostic tool to determine real time operation of the receiver block. If your model is not running in real time, you can:

  • Use Burst mode

  • Use vector-based processing

  • Accelerate with code generation

Either or both of these techniques may be applied to approach or achieve real-time performance.

SDRu Receiver Block

On the block mask, select the Show lost samples port block parameter.

This enables the lost samples port so that the block outputs a value to indicate if one or more packets is dropped during host reception of USRP® hardware data.

  • Zero indicates no data loss

  • A positive number indicates that overruns occurred

This port is a useful diagnostic tool to determine real time operation of the receiver block. If your model is not running in real time, you can:

  • Use Burst mode

  • Tune model

  • Run Accelerator

Either or both of these techniques may be applied to approach or achieve real-time performance.

Using Burst Mode

Use burst mode when your model is experiencing underruns or overruns because it can't keep up with the amount of data being sent or received in real time. Burst mode allows you to buffer a minimum set of contiguous samples without underruns or overruns. Note, however, that between bursts, overruns and underruns can still happen, especially with large burst sizes. If your model can keep up in real time, then do not use this feature.

The maximum burst size (in frames) is imposed by the operating system and the USRP® device UHDTM. The maximum size imposed by the UHDTM is approximately 1GB, or 256 megasamples. This max number of samples is enforced by our software. With a frame size of 4000, for example, the max number is approximately 67k frames. Depending on the specific host, a lower limit may hold due to memory constraints. Exceeding the limit will be flagged by an "unable to allocate memory" error.

The default number of frames in a burst is 100.
